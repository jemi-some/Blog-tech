
안녕하세요! 오늘은 제가 꼬박 3시간 동안 엉덩이 한 번 떼지 않고 몰입해서 완성한 '블로그 포스팅 자동화 시스템' 구축기를 공유합니다.

### 🧐 왜 시작했나?
처음엔 개발 과정을 기록으로 남기기 위해 블로그를 시작할 생각이었습니다. 예전에 사용하던 GitHub 블로그를 다시 쓸까 고민도 했지만, 커스텀 자유도가 너무 높은 탓에 글을 쓰는 기록보다는 UI 디자인을 수정하는 데 에너지를 더 쏟게 될 것 같아 포기했습니다.

그러던 중 Hashnode를 알게 되었습니다. 개발자 친화적인 플랫폼이면서도 UI가 이미 충분히 깔끔해 콘텐츠에만 집중할 수 있다는 점이 매력적이었습니다. 다만, GitHub 블로그처럼 **'.md 파일 작성 후 push하면 자동으로 포스팅되는 워크플로우'**는 포기하고 싶지 않았습니다.

그래서 평소 공부하던 Webhook 개념도 익히고, 개발자들 사이에서 핫한 자동화 툴인 n8n을 활용해 직접 나만의 자동 포스팅 시스템을 구축해보기로 했습니다.

### 🛠️ 자동화 워크플로우 설계

1. 작성 (VS Code/Google Antigravity): .md 파일로 TIL 내용을 작성합니다.

2. AI 첨삭: 사이드바의 에이전트를 활용해 AI 첨삭을 받습니다.

3. 배포 트리거 (GitHub Push): 최종 파일을 메인 브랜치에 push 합니다.

4. 블로그 업로드 (n8n): n8n이 GitHub의 새로운 커밋을 감지하여 Hashnode API로 글을 전송합니다.

### 🤖 "AI가 있으니 금방 n8n 사용법을 익히겠지"의 반전
n8n에도 자체 n8n AI가 있어서 설계도 도와주고 빌드까지 해주는 기능이 존재합니다. 그래서 금방 끝날 줄 알았는데, 각 노드의 세부 설정은 처음이라 공식 문서를 뒤져가며 Gemini에게 과외를 받기 시작했습니다.

AI: "자, 노드는 이렇게 연결하시고요, 코드는 이걸 쓰세요."

나: "Credential(인증) 설정은 어떻게 하는 건데?"

AI: "아, 그건 수작업으로 직접 하셔야 합니다."

나: (결국 직접 문서 뒤져가며 API 키 넣고 헤더 설정 중...)

결국 AI가 가이드를 주면, 제가 멱살 잡고 세부 에러를 고치며 완성하는 기묘한 협업이 이어졌습니다. 3시간의 사투 끝에 탄생한 이 시스템은 저의 두 번째 자동화 시스템이 되었습니다. (첫 번째는 구글 앱스 스크립트를 활용한 자동화 시스템이었습니다.)

### 🛠️ 시스템 구조 (Logic Flow)
설계한 워크플로우는 다음과 같습니다:

- GitHub Webhook: 깃허브에 push 이벤트가 발생하면 신호를 받습니다.

- Filter (.md): 여러 파일 중 마크다운 파일만 골라냅니다.

- Check Existing Post: Hashnode API를 조회해서 "글을 발행하는 건지, 업데이트 하는 건지" 확인합니다.

- Branching (IF):

    - 글이 있으면? -> UpdatePost (기존 글 수정)

    - 글이 없으면? -> PublishPost (새 글 발행)


### 🫠 삽질했던 트러블슈팅 포인트

1. GitHub Webhook 연동을 위한 OAuth 앱 등록

    문제: n8n이 깃허브의 신호를 받으려면 정식 인증이 필요했습니다.

    해결: GitHub Developer settings에서 OAuth 앱을 생성하고, n8n에 Credential을 수동으로 등록하여 보안 연결을 성공시켰습니다.

2. Filter 노드에서 수식 넣기

    해결: 

    ```JavaScript
    {{ ($json['body.commits'].modified[0] || $json['body.commits'].added[0] || "").endsWith('.md') }}
    ```

3. 한글이 다 깨져요! (Base64 Decoding)

    문제: GitHub API로 가져온 본문 데이터는 Base64로 인코딩되어 있어, 그대로 보내면 한글이 깨졌습니다.

    해결: Prepare Hashnode Data 노드에서 아래 수식을 사용하여 UTF-8로 완벽하게 복구했습니다.

    ```JavaScript
    {{ $json.content.base64Decode() }}
    ```

4. 중복 업로드 방지 (조회 노드 추가)

    문제: 단순히 Publish 노드만 두면, 오타 하나 고쳐서 push 할 때마다 똑같은 새 글이 무한 생성되는 대참사가 예상되었습니다.

    해결: Check Existing Post 노드를 중간에 삽입하여 **"이미 있으면 Update, 없으면 Create"**라는 분기(Branching)를 추가했습니다.

5. "Variable $host was not provided" (GraphQL의 늪)

    문제: Hashnode API 조회 쿼리를 날릴 때 필수 변수가 누락되었다는 에러가 반복되었습니다.

    해결: Hashnode에 업로드하는 노드를 만들 때는 publicationId가 필요했는데, 여기서는 host를 사용해야 했습니다. 

6. JSON 문법 오류와 JSON.stringify의 구원

    문제: 마지막 Publish 노드에서 자꾸 JSON 형식이 잘못되었다는 에러가 발생했습니다. 알고 보니 본문(Markdown)에 포함된 줄바꿈, 큰따옴표, 특수문자들이 JSON의 구조를 깨뜨리고 있었습니다.

    해결: 데이터를 보낼 때 단순히 변수만 넣지 않고, 아래와 같이 **JSON.stringify()**를 사용하여 데이터를 안전하게 '문자열화'하여 해결했습니다.

    ```JavaScript
    "contentMarkdown": {{ JSON.stringify($('Prepare Hashnode Data').item.json.contentMarkdown) }}
    ```

    교훈: 텍스트 데이터가 길고 복잡할수록, 시스템이 오해하지 않도록 **규격화해서 포장(Stringify)**해주는 과정이 필수라는 것을 배웠습니다.


### 🏆 마치며

앞으로도 실제 사용하면서 끊임없이 수정하고 개선해 나가야 하겠지만, 일단은 제 손으로 직접 만든 첫 번째 파이프라인을 완성했다는 것에 큰 만족감을 느낍니다!
